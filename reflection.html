Task 3

Reflection on Skill Development

3.1 Academic Skills

Critical Thinking and Analysis

The most demanding analytical challenge was deadlock prevention in Part 1. Deadlock doesn't surface as a runtime error — it manifests as a program that silently hangs, with no exception or stack trace to guide debugging. Reasoning from first principles, deadlock requires four simultaneous conditions (Coffman et al., 1971), and eliminating any one is sufficient; I chose to break the circular-wait condition via a global lock-ordering protocol. Similarly, choosing Visitor for the analytics artefact required weighing two competing OCP directions — the element hierarchy versus the operation set — and documenting that trade-off explicitly in code comments, rather than treating patterns as universally beneficial.

Ethical Considerations in AI-Driven Object-Oriented Design

Artefact 8 raises a real ethical dimension: wrapping an AI classifier behind an abstraction hides not only the provider but also the model's behaviour — its training data, known biases, failure modes. A poorly designed class hierarchy can propagate bias invisibly; if a ContentModerationService silently uses a model more likely to flag certain languages or dialects as unsafe, the OO design has made that bias harder to discover, not easier. Transparency and auditability are therefore design concerns, not just operational ones — the audit log in PricingEngine and the structured output of the Visitor pattern both support explainability, and every AI call in the portfolio records enough context for post-hoc review, deliberately avoiding the "black-box coupling" anti-pattern. This extends to testability: a class whose AI dependency can't be mocked can't be verified reliably in production — not merely inconvenient, but an ethical failure in safety-critical applications.

Problem-Solving

A recurring approach was using abstraction to defer decisions. Designing the AI provider-switching system, I initially considered a concrete OpenAI-only implementation; recognising that vendor lock-in is a real risk in AI projects — providers change APIs, pricing, availability — I refactored to introduce the AIServiceFactory abstraction before writing any concrete code. Applied consistently, this "abstract first" heuristic produced a more extensible final design.

Research and Self-Study

The module required engagement with primary sources beyond lecture materials: NIST SP 800-63B (Grassi et al., 2017) informed password-hashing parameters; Coffman et al. (1971) provided the formal deadlock conditions used to justify lock ordering; GoF pattern descriptions (Gamma et al., 1994) were consulted to ensure implementations matched canonical intent, particularly the Visitor's double-dispatch mechanism.

3.2 Employability Skills

Time Management

6 artefacts across two parts, each needing implementation, testing, and documentation. I adopted a fixed priority order — implementation first (ensures executability), tests second (validates correctness), documentation last (accurately reflects what was built) — preventing the common failure of documenting intended behaviour that was never implemented.

Communication and Literacy

Every artefact includes structured docstrings explaining the chosen pattern, its benefits, and its acknowledged trade-offs — reflecting that code is read far more often than written, and that a future maintainer who understands why RLock beats Lock is more likely to preserve that property when extending the system.

Resilience and Interpersonal Skills

The concurrent test suite for the banking system initially produced intermittent failures traced to a subtle race condition in the test setup code itself, not the production code. Working through this required resilience to question my own assumption about which layer contained the bug, and the discipline to write a minimal reproducible case before attempting a fix.
