Week 9

ShopEase Architecture

The ShopEase Architecture is designed as a scalable, modular and secure e-commerce system using a layered object-oriented architecture. The code separates the application into three main layers: the Presentation Layer, the Business Logic Layer and the Data Access Layer. This structure supports maintainability because each layer has a distinct responsibility, and changes in one part of the system can be made with minimal impact on the others.

Presentation

Represented by the ShopEaseCLI class. this class simulates the user interface of the system, such as a web interface or mobile application. Its role is to handle user interaction and pass user requests to the relevant business services. it does not contain the main business rules itself, which helps maintain separation concerns.

Business Logic

The AuthService manages user registration and login. The CatalogueService manages product listings and stock. The OrderService controls the order lifecycle, including order creation, payment processing and status updates. The PaymentGateway abstraction supports flexible payment processing, while the EventBus supports notification-style behaviour using the Observer Pattern.

Data Access

The Data Access Layer is represented by repository classes: UserRepository, ProductRepository, OrderRepository. These repositories manage storage and retrieval of users, products and orders. In this implementation, the repositories use in-memory storage, but in the design allows them to be replaced later with database-backed repositories. This supports scalability and maintainability because the business services do not need to know the details of how the data is stored.

Security Features

Security is addressed mainly through the authentication design. The code includes password hashing using PBKDF2-HMAC-SHA256 with a salt. it also uses hmac.compare_digest for password verification, which supports constant-time comparison and helps reduce exposure to timing attacks.

Design Patterns Implemented

Observer Pattern: demonstrated by the EventBus by allowing parts of the system to subscibe and respond to events. This could support notifications when an order is placed or shipped.

Strategy Pattern: demonstrated by the PaymentGateway andMockPaymentGateway implementations. this allows different payment methods to be introduced without changing the OrderService. For example future implementations could include card payments, PayPal or other payment providers.

Repository Pattern: repository classes separate data access from business logic. This improves maintainability because storage can be changed without rewriting the services.

Dependency Injection: demonstrated by the build_shopease() function which wires together repositories, services, the eventbus and payment gateway. This shows Dependency Injection, because dependencies are passed into components rather than being created internally. This reduces coupling and makes the system easier to test.

Reflection

Week 9 Reflection

Unit 9 brought together many of the concepts covered throughout the module and demonstrated how object-oriented principles, design patterns, and secure coding practices contribute to effective software architecture. Through the lecturecast and reading materials, I developed a greater understanding of architectural styles such as layered architectures, monolithic applications, and microservices, along with the advantages and trade-offs associated with each approach.

The ShopEase case study was particularly valuable because it provided a practical example of how a large-scale system can be designed to meet requirements for scalability, maintainability, security, and extensibility. Analysing the layered architecture helped me appreciate the importance of separating responsibilities between the presentation, business logic, and data access layers. I recognised how this separation improves maintainability by reducing coupling between components and allowing individual parts of the system to evolve independently.

The unit also reinforced the role of object-oriented design in building scalable and secure systems. Concepts such as encapsulation, abstraction, dependency injection, and modular design were shown to be architectural as well as programming concerns.

Outcome

Unit Outcome

The unit highlighted that there is rarely a single perfect architecture; instead, architects must balance scalability, maintainability, performance, and security requirements based on business needs. This understanding will be valuable when evaluating technology solutions and participating in future design decisions.

Developing ShopEase allowed me to apply architectural principles in a practical context, demonstrating how clear separation of responsibilities improves maintainability, scalability, security and testability.
