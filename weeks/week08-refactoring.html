Week 8

Tasks Done

  • Studied what code smells are, how to detect them, and why they signal deeper design issues rather than being problems in themselves.
  • Reviewed refactoring techniques for improving maintainability and readability without changing external behaviour.
  • Refactored existing code on the SOLID shopping system and the Dependency Injection to eliminate specific, named code smells using object-oriented principles.
Reflection

Week 8 Reflection

The most useful shift this week was learning to name code smells precisely rather than reaching for a vague "this feels messy." "God Class" and "hidden dependency" are more actionable diagnoses than "this class does too much," because the name points directly at the refactoring technique that fixes it — extract responsibility, or invert the dependency. Revisiting artefacts I'd already built for earlier weeks, specifically to identify what smell each "before" version demonstrates, made the connection between diagnosis and technique much more concrete than analysing unfamiliar example code would have.

The harder lesson was recognising that not every long class or every direct instantiation is actually a smell; context matters. A five-line utility class instantiating a simple value object internally isn't a hidden-dependency problem; the smell is specifically about dependencies that need to vary or be tested in isolation. Learning to tell the difference is what separates mechanically applying refactoring patterns from actually improving a design.

Outcome

Unit Outcome

By completing this week's exercises, I was be able to:

  • Identify specific, named code smells in a codebase rather than relying on vague impressions of "messy" code.
  • Apply targeted refactoring techniques: extracting responsibility, inverting dependencies that address the diagnosed smell directly.
  • Use an existing test suite to verify that a refactor preserves behaviour rather than silently changing it.
  • Distinguish genuine code smells from superficially similar code that doesn't actually need refactoring.
