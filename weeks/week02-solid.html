Week 2

Tasks Done

  • Studied and reviewed the five SOLID principles and their role in object-oriented software design
  • Analysed a poorly designed online shopping system and identified violations of SRP, OCP and DIP.
  • Refactored an existing, poorly-designed Order class (an online shopping system) so the result adheres to all five SOLID principles.
  • Tested and evaluated the refactored solution to demonstrate improved maintainability, scalability and flexibility.
Practice

SOLID Principles — Online Shopping System

The exercise involved analysing and refactoring a poorly designed online shopping system. The original implementation placed cart management, total calculation and payment processing within a single Order class, violating the Single Responsibility Principle. In my refactored solution, I separated these concerns into dedicated classes such as Product, Order, DiscountStrategy and PaymentMethod. This improved cohesion and made the system easier to maintain and extend.

S
Single Responsibility
Order manages items only; PaymentMethod handles payments; DiscountStrategy applies discounts.
O
Open / Closed
New payment methods (e.g. CryptoPayment) added without modifying existing classes.
L
Liskov Substitution
CreditCard, PayPal, and Crypto all extend PaymentMethod and are interchangeable.
I
Interface Segregation
PaymentMethod and DiscountStrategy are focused single-method abstractions.
D
Dependency Inversion
Order depends on abstractions, not on EmailService or concrete payment classes.
# Before: SRP / OCP / DIP violations
class Order:
    def pay(self, payment_type):
        if payment_type == "credit": ...   # ← must edit to add new method
        elif payment_type == "paypal": ...

# After: SOLID compliant
class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount: float) -> bool: ...

class CryptoPayment(PaymentMethod):          # new — zero changes elsewhere
    def pay(self, amount: float) -> bool: ...

class Order:
    def __init__(self, discount: DiscountStrategy, payment: PaymentMethod): ...

Reflection

Week 2 Reflection

In Unit 2, I developed my understanding of the SOLID principles and how they contribute to creating maintainable, scalable and adaptable object-oriented software. Building on the OOP concepts introduced in Unit 1, I learnt that effective software design is not only about functionality but also about reducing complexity and supporting future changes (Marcotte and Zebdi, 2022)

Refactoring Order made the five principles feel far less abstract than reading their definitions. The clearest moment was realising that SRP, OCP, and DIP were all being violated by the same three lines of code — the if/elif on payment_type is simultaneously a class doing too much, a design closed to extension, and a dependency on a string instead of an abstraction. Seeing one smell manifest as three separate principle violations made the principles feel connected rather than like five independent rules to memorise.

The Liskov Substitution Principle was the one I initially underestimated. It's easy to assume that "subclasses share a base class" automatically satisfies LSP, but the real test is whether every caller of PaymentMethod.pay() behaves correctly no matter which concrete class is passed in, which only becomes obvious once you write a test that constructs an Order with each payment type interchangeably and asserts the same contract holds for all three.

Outcome

Unit Outcome

Overall, this unit showed how the SOLID principles can improve software quality by making systems easier to understand, test and modify. The shopping system exercise provided practical experience in applying these principles and reinforced the importance of good design decisions in professional software development.
