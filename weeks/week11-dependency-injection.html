IoC Container

To further demonstrate IoC, the artefact includes a lightweight DIContainer responsible for configuring and supplying application dependencies. This moves responsibility for object creation away from the business logic and centralises dependency management. The design also includes a MultiChannelService, which applies a composite-style approach to deliver notifications through multiple channels simultaneously.

The lightweight DIContainer class registers service factories and resolves them on demand, supporting singleton and transient lifetimes, replicating the core behaviour of dependency-injector without any external packages.

# Before: tightly coupled
class UserManager:
    def __init__(self):
        self.notifier = EmailService()   # ← hidden dependency, untestable

# After:constructor injection
class UserManager:
    def __init__(self, notifier: NotificationService):
        self._notifier = notifier           # ← injected; swap without editing class

# Test: no real email sent
def test_register_sends_welcome_notification(self):
    mock = MockNotificationService()
    mgr  = UserManager(notifier=mock)   # inject mock
    mgr.register_user("alice")
    self.assertEqual(len(mock.calls), 1)
    self.assertIn("Welcome", mock.calls[0][1])

Reflection

Week 11 Reflection

This unit strengthened my understanding of how Dependency Injection (DI) and Inversion of Control (IoC) support the development of flexible, maintainable, and testable software systems. Using the Dependency Injection artefact, I explored how tightly coupled code can create challenges for maintenance and testing, and how these issues can be addressed by introducing abstractions and injecting dependencies rather than creating them directly within classes.

A particularly valuable lesson was understanding how dependency injection improves testability. By injecting a mock notification service, I was able to test the behaviour of UserManager in isolation without triggering real external services. This reinforced the importance of designing software that can be validated through automated testing, a concept that I later applied in my Capstone Project when developing maintainable and scalable software components.

By completing this week's exercises, I was able to:
