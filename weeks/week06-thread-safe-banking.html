BankAccount — thread-safe operations

All public methods acquire an RLock before mutating state. An RLock is used instead of a plain Lock to allow the same thread to re-enter locked methods without self-deadlock.

# banking_system.py — core excerpt
class BankAccount:
    def __init__(self, account_number: str, initial_balance: float = 0.0):
        self.account_number = account_number
        self._balance       = initial_balance
        self._lock          = threading.RLock()   # reentrant — safe for nested calls
        self._transaction_log: list[dict] = []

    def deposit(self, amount: float) -> float:
        self._validate_amount(amount)
        with self._lock:                          # atomic read-modify-write
            self._balance += amount
            self._log_transaction("deposit", amount, self._balance)
            return self._balance

    def withdraw(self, amount: float) -> float:
        self._validate_amount(amount)
        with self._lock:
            if amount > self._balance:
                raise InsufficientFundsError(...)
            self._balance -= amount
            return self._balance

Deadlock prevention: Ordered lock acquisition

The transfer() function always acquires the two account locks in a globally consistent order (by account_number). This eliminates the circular-wait condition required for deadlock.

def transfer(source: BankAccount, destination: BankAccount, amount: float):
    # Sort by account_number → always acquire in the same order
    first, second = (
        (source, destination) if source.account_number < destination.account_number
        else (destination, source)
    )
    with first._lock:
        with second._lock:           # both held — transfer is atomic
            source._balance   -= amount
            destination._balance += amount

Reflection

Week 6 Reflection

This week was the clearest demonstration so far that code which passes every sequential test can still be broken under concurrent access. Choosing a reentrant lock over a plain lock mattered specifically because internal methods call one another: a plain lock would self-deadlock the moment one locked method called another on the same thread, a failure mode that would never surface in a single-threaded test.

Deadlock prevention was the more conceptually demanding part: reasoning about two threads each trying to acquire the same two locks in opposite orders required thinking about the system's behaviour as a whole, not just one account's operations in isolation. Always acquiring locks in a fixed, consistent order (sorted by account identifier, regardless of which account initiated the transfer) removes the circular wait that deadlock depends on — a small rule with a disproportionately large effect on correctness. The concurrent load test, spinning up dozens of threads and checking that total money is conserved afterwards, was the only way to gain real confidence that this held in practice rather than just in theory.
