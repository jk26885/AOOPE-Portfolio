Task 1

Summarise Key Learning Outcomes

A key learning outcome was understanding that advanced object-oriented programming is not only about creating classes, but about assigning responsibilities clearly. This was shown in the SOLID shopping artefact, where a monolithic order class was refactored into smaller, focused classes with clearer responsibilities. This improved maintainability, extensibility and testability. The use of abstraction, encapsulation, inheritance, polymorphism and dependency inversion helped me understand how structured object-oriented code can reduce coupling and support future change. Design patterns are useful in this context because they provide reusable solutions to common software design problems and create a shared language for developers (Refactoring Guru, 2026).

Secure coding was another important area of development. In the ShopEase and e-learning artefacts, I applied password hashing, role-based access control, validation and secure authentication logic. The use of PBKDF2-HMAC-SHA256 with a random salt showed how security can be designed into the system from the beginning. OWASP recommends that passwords should not be stored in plain text and should instead be protected using strong hashing algorithms with unique salts, including PBKDF2 where appropriate (OWASP, 2026). Python also recommends using compare_digest() rather than standard equality checks when comparing cryptographic digests, as this reduces exposure to timing attacks (Python Software Foundation, 2026).

Password hashing and constant-time comparison, designed in from the start (artifact2_shopease.py):

def hash_password(password: str, salt: bytes | None = None) -> tuple[str, str]:
    if salt is None:
        salt = os.urandom(16)
    key = hashlib.pbkdf2_hmac("sha256", password.encode(), salt, 260_000)
    return key.hex(), salt.hex()

def verify_password(password: str, stored_hash: str, stored_salt: str) -> bool:
    """Constant-time comparison to prevent timing attacks."""
    salt = bytes.fromhex(stored_salt)
    candidate, _ = hash_password(password, salt)
    return hmac.compare_digest(candidate, stored_hash)

The thread-safe banking system helped me understand concurrency and robustness. It used re-entrant locks and ordered lock acquisition to reduce the risk of deadlock during account transfers. This was one of the more challenging parts of the module because concurrent programming requires thinking beyond straightforward sequential execution. I learned that reliability in concurrent systems depends not only on whether code works in a simple test, but also on whether it behaves correctly when multiple operations happen at the same time.

class BankAccount:
    def __init__(self, account_number: str, initial_balance: float = 0.0):
        self.account_number = account_number
        self._balance = initial_balance
        # RLock allows the same thread to acquire the lock multiple times,
        # preventing self-deadlock in methods that call each other.
        self._lock = threading.RLock()

def transfer(source: BankAccount, destination: BankAccount, amount: float) -> None:
    # Determine a consistent lock-acquisition order to avoid deadlock
    first, second = (
        (source, destination)
        if source.account_number < destination.account_number
        else (destination, source)
    )

    with first._lock:
        with second._lock:
            source._balance -= amount
            destination._balance += amount

Critical Commentary

The selected artefacts demonstrate the application of key object-oriented programming principles, software architecture concepts and testing practices developed throughout the module. The SOLID shopping system artefact shows how SOLID principles can improve maintainability and extensibility by separating responsibilities and introducing abstractions for discounts and payment methods. This reduces coupling and makes the system easier to extend without modifying existing code.

The Dependency Injection and IoC artefact reinforce the importance of loose coupling through Dependency Injection and Inversion of Control. By depending on abstractions rather than concrete implementations, the design improved flexibility, testability and maintainability. The use of mock services also demonstrated how architectural decisions can simplify automated testing and reduce reliance on external dependencies.

A major strength of the portfolio was its focus on testing. The elearning.py artefact applied Test-Driven Development (TDD), using the Red-Green-Refactor cycle to guide implementation. The banking_system.py and test_banking_system.py artefacts further highlighted the challenges of concurrent programming, including thread safety and deadlock prevention, while demonstrating the importance of comprehensive testing for validating behaviour under concurrent conditions. The testing_mocking.py artefact expanded this understanding by using mocking techniques to isolate dependencies and enable reliable, deterministic tests.

A key lesson learned was the need to balance abstraction against complexity. While SOLID principles, Dependency Injection and extensive testing improved maintainability and software quality, they also introduced additional classes and architectural overhead. This reinforced that design patterns and abstractions should be applied only when they solve genuine maintainability, scalability or testing challenges rather than simply increasing design complexity. Overall, these artefacts strengthened my understanding of how object-oriented principles, software architecture and testing practices work together to produce reliable and maintainable software systems.

Reflect on Skill Development

Academically, the module strengthened my critical thinking by requiring me to evaluate design options and justify architectural decisions. For example, selecting appropriate design patterns and applying dependency injection required consideration of maintainability, testability and future extensibility. My learning progressed through Bloom's Revised Taxonomy, moving from understanding object-oriented concepts to applying, analysing, evaluating and creating software solutions (Anderson and Krathwohl, 2001; eLearning Industry, 2026).

The module also increased my awareness of ethical and responsible software design, particularly in AI-enabled systems. I learned that AI components should be designed to be testable, explainable and auditable, rather than treated as opaque black boxes, ensuring that limitations and potential biases are understood and managed appropriately.

From an employability perspective, the module strengthened my time management, technical communication and resilience. Developing multiple artefacts required careful planning and prioritisation to balance implementation, testing and documentation. Writing code comments and reflective evaluations also improved my ability to communicate technical concepts clearly, explaining not only how the code works but also the reasoning behind key design decisions. These experiences helped me develop skills that are directly transferable to professional software development environments, where effective communication, organisation, and adaptability are essential.

Reflective Model: Johari Window

The Johari Window provides a useful perspective on my development. My open area expanded as my understanding of object-oriented design, secure programming and testing became visible through my artefacts and reflections. My blind area reduced through feedback, module materials, testing results and self-evaluation. My hidden area decreased because reflective writing encouraged me to explain my reasoning and assumptions more clearly. The unknown area also became more visible, especially in relation to asynchronous programming, AI lifecycle management and advanced testing. The Johari Window supports this type of reflection by encouraging self-awareness across open, blind, hidden and unknown areas (Luft and Ingham, 1955; Creately, 2026).

Reflective Model: Gibbs' Reflective Cycle

Gibbs' Reflective Cycle also helped me structure my learning. The module involved repeated stages of design, implementation, testing, evaluation and action planning. Initially, I found concurrency and AI-adaptable architecture challenging, but reflection showed that testing, interfaces and dependency injection were the most valuable techniques for improving reliability. Gibbs' model supports structured reflection through description, feelings, evaluation, analysis, conclusion and action planning (Gibbs, 1988; University of Edinburgh, n.d.).
